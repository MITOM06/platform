### 주의사항
- Token ops are keyed by the bot's **ownerUserId** (the user the bridge session belongs to), not the admin's id — `issue/revoke/listSessions(bot.ownerUserId, bot.botUserId)`. This matches the connector-service session model `{userId, botUserId}`.
- Token is held only in the dialog's build scope and never persisted (Riverpod state holds only the session summary, which has no token).
- Dialog dismiss uses the existing `actionCancel` key (no generic `commonClose` key exists in the ARB); all 9 new botAdmin keys match the web plan's parity list. The web `botAdmin.*` group is being added in parallel by web-dev.
- `GET /api/admin/external-bots` (chat-service) is added in parallel by backend-dev; the repo defensively coerces a non-list response to `[]` (mirrors web's `Array.isArray` guard). Revoke button only shows when an active session exists for that bot.
