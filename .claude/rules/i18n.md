# i18n — Multi-Language Rules (Flutter client)

The app supports 7 languages: **en, vi, zh, ja, ko, es, fr**. The user selects the language in
Settings → the choice is persisted (`shared_preferences`) and applied to the entire UI.

## Mandatory Rules

1. **NO hardcoded UI strings** in widgets. Every text must be retrieved from
   `context.l10n.<key>` (extension in `lib/core/l10n/l10n_ext.dart`).
2. **Template** is `lib/l10n/app_en.arb` — add new keys here first (with `@key`
   metadata if there are placeholders like `{email}`, `{count}`).
3. **Every new UI element ⇒ add keys to ALL 7 files** `lib/l10n/app_*.arb`. Missing keys in
   a locale fallback to English (still builds, but should be avoided).
4. Name keys using camelCase and functional groups: `loginTitle`, `valEmailRequired`,
   `chatDefaultTitle`, `groupName`… Avoid duplicate meanings.
5. After modifying ARB: `flutter gen-l10n` (or `flutter pub get`) regenerates
   `lib/l10n/app_localizations*.dart`. **DO NOT manually edit** generated files.
6. Strings with variables must use ARB placeholders (`"resendIn": "Resend in {seconds}s"`),
   do NOT concatenate strings manually.
7. Format date/time/numbers according to the locale via `package:intl` (`DateFormat`,
   `NumberFormat`), do not hardcode formatting patterns.

## Architecture

- `main.dart`: `MaterialApp.router` uses `locale`, `AppLocalizations.supportedLocales`,
  `AppLocalizations.localizationsDelegates`.
- `lib/core/providers/locale_provider.dart`: `LocaleNotifier` (Riverpod) stores choices;
  `null` = system language. `kSupportedLocales` + `kLanguageNames` is the source of truth for the language list and native names.
- Configuration: `l10n.yaml` (arb-dir `lib/l10n`, output-class `AppLocalizations`).

## Backend
Localization is client-side. The auth-service/chat-service should return English messages/errors or error codes; the client maps them to localized strings. No backend translation is needed at this stage.
